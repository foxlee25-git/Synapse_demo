Customer-to-Agent System

先拿客户数据，让 AI 理解客户，再化身不同 Agent 去解决、安排和执行工作。

Synapse 的产品端不是先从“空白聊天框”开始，而是先把客户的文档、CRM、聊天记录、订单、广告和服务数据接进来，形成客户理解层。系统再根据客户当前情况自动调用不同 Agent，去分析问题、安排动作、推动执行。

先吃进客户数据

CRM、广告、订单、客服、文档和 SOP 被统一接入，不让 Agent 在空上下文里工作。

Understand

Execute

Customer Understanding Layer

产品的第一层不是 Agent，而是“客户理解层”

只有先拿到客户的数据，并让 AI 理解客户的业务状态、问题和优先级，后面的 Agent 才能不是瞎干活，而是真的围绕客户情况去工作。

Step 01

先接入客户数据

导入客户的 CRM、订单、广告账户、客服记录、聊天记录、文档和 SOP，把原本分散的信息统一拉进来。

Step 02

AI 先理解客户情况

系统先形成客户画像、业务摘要、当前异常、潜在机会和优先级，而不是马上把任务丢给单个 Agent。

Step 03

再化身不同 Agent

根据客户当前状态，自动拆分成 Growth、Sales、Knowledge、Support 等不同角色的 Agent 去各自处理。

Step 04

解决、安排、执行

有的 Agent 负责分析问题，有的 Agent 负责安排流程，有的 Agent 负责直接执行动作，并在关键节点拉人确认。

How It Works

先理解客户，再在群聊里分给 Agent

  • 系统先形成客户摘要、问题地图和优先级
  • @ 某个 Agent 后，系统自动把客户上下文一起带过去
  • 执行状态回写到群聊，同时同步到工作台、审批流和任务面板

Assignment Rules

适合在群聊里分给 Agent 的工作

@Growth Agent

广告复盘、预算建议、异常定位、增长摘要

@Sales Agent

线索排序、客户摘要、跟进建议、老板介入提醒

@Knowledge Agent

FAQ、SOP 更新、客服知识整理、文档归档

Why It Matters

协作方式更像“带 Agent 的团队群”

谁发现问题，谁就在群里点名 Agent 去处理；谁需要结果，谁在同一个线程里拿到交付、进度和审批请求。这样比切来切去找控制台自然得多。

Task Flow

客户数据如何变成多个 Agent 的执行流程

1
接入客户数据

CRM、广告、客服、销售报表、知识库与聊天记录完成同步。

2
理解客户情况

系统生成客户画像、问题摘要、优先级和建议派单结构。

3
分化成不同 Agent

Growth Agent 处理广告异常，Sales Agent 跟进线索，Knowledge Agent 更新 FAQ 与 SOP。

4
执行、回报、审批

各 Agent 把结果回写到群聊和工作台，关键动作等待人工确认后正式下发。
