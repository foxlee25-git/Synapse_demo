Product Page Architecture

How the Shadow Network product should be organized.

This page turns the product idea into a concrete page map. It shows the core surfaces, the user path between them, and what each page is responsible for in the first real version of Synapse.

Open Architecture Map See Dashboard See Chat
Core Pages Dashboard / Chat / Agents / Data
Support Pages Outputs / Settings
Main Loop Data -> Agent -> Chat -> Dashboard

Main Flow

Employees talk to one Shadow Network. The product routes them through the right surfaces.

Upload context Configure role shadows Work in chat Track in dashboard

Top Navigation

Primary product surfaces

  • Dashboard: overview of active agents, recent outputs, pending reviews, and business signals.
  • Chat: the main Shadow Network workspace where employees ask for work to be done.
  • Agents: role-based shadow configuration with scopes, templates, and restrictions.
  • Data: file upload, connected sources, and agent-to-data mapping.
  • Settings: governance, routing, budget, and access rules.

Secondary Surfaces

What comes right after MVP

  • Outputs: saved summaries, briefs, email drafts, and task bundles generated by the network.
  • Tasks: optional execution board once chat outputs need stronger workflow tracking.
  • Approvals: optional review center for higher-risk actions or shared publishing flows.

Recommended User Journey

The intended movement across pages

MVP path
Step 1: Data

The team uploads docs, exports, SOPs, sheets, and connects lightweight systems.

Step 2: Agents

Admin or founder binds the right sources to Sales, Assistant, Support, or custom shadows.

Step 3: Chat

Employees talk to the Shadow Network to get summaries, drafts, recommendations, and next steps.

Step 4: Dashboard

The business sees what was generated, what is active, and what still needs a human decision.

MVP Rule

Do not overbuild

The first version only needs the four core pages to be useful and sellable.

Design Rule

One entry in front

Employees should feel like they are talking to one network, not navigating a maze of disconnected tools.

Scaling Rule

Add depth by layer

Outputs, tasks, and approvals can come after the core loop proves that the network saves time and reduces coordination cost.
